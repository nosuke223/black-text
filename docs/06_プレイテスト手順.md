# 06_プレイテスト手順

## 1. 目的

本手順は、固定文章版の`SCENARIO-001 三度目の呼び声`を少人数でプレイテストし、ゲームルール、固定文章、UI、計測、実行環境の問題を分けて確認するために使用する。

段階Aでは、AIによる文章生成の品質は評価対象に含めない。プレイヤーへ法則、正解経路、選択肢の内部分類を事前に説明しない。

初回の少数プレイでは合格率などの数値目標を置かず、発生した問題の種類と原因を収集する。

---

## 2. 準備

進行役は、開始前に次を用意する。

- テスト対象の公開URL
- 対象ビルドのGitコミットID
- PCと対応ブラウザの最新版
- プレイヤーごとの匿名ID（例：`PT-001`）
- `docs/07_プレイテスト記録票.md`のコピー
- JSONと聞き取り記録を保存する、Git管理外の保管場所

プレイヤーの氏名、メールアドレスなど、テストに不要な個人情報は記録しない。匿名IDと連絡先を対応付ける必要がある場合は、プロジェクト資料およびJSONとは別に管理する。

---

## 3. プレイヤーへ伝える手順

進行役は、原則として次の内容だけを伝える。

1. URLをPCブラウザで開く
2. 「はじめる」を押し、自分の判断で最後まで進める
3. 結果が出たら「記録を見る」を押す
4. 「プレイテスト記録を保存」を押し、JSONファイルを保存する
5. もう一度試したい場合は、JSON保存後に「もう一度試す」を押す
6. 各プレイの終了後に、保存したJSONファイルを進行役へ渡す

再挑戦を開始すると、ブラウザメモリ上の前回記録は新しいプレイ用に初期化される。前回のJSONを保存してから再挑戦する。

プレイ中は、文章を読み返してもよい。進行役へ質問してもよいが、操作不能などの環境問題を除き、進行役は法則や安全な選択を説明しない。

---

## 4. 進行役の観察

進行役はプレイを誘導せず、次の事実を記録する。

- 呼びかけの回数を数えている様子があったか
- 過去の文章を読み返そうとしたか
- どの選択肢で長く迷ったか
- 鈴の利点または代償を意識した発言があったか
- 結果を見た直後に、原因についてどのような発言をしたか
- 記録画面とJSON保存を説明なしで完了できたか
- 再挑戦したいと自発的に感じたか
- 操作不能、表示崩れ、保存失敗が起きたか

選択時間、選択履歴、手掛かり、結果、総プレイ時間はJSONから確認する。観察記録へ同じ値を手作業で転記しない。

---

## 5. プレイ後の聞き取り

最初のプレイが終わるまで、次の質問を見せない。

1. この怪異には、どのような法則があると思いましたか
2. そう考えた文章や出来事はどれですか
3. 呼びかけの回数に意味があると感じましたか。いつ気づきましたか
4. 鈴にはどのような利点と代償があると思いましたか
5. 今回の結果になった理由を、どのように考えていますか
6. 理不尽、分かりにくい、または情報不足だと感じた箇所はありましたか
7. 怖かった箇所、または選ぶのをためらった箇所はありましたか
8. もう一度試したいと思いましたか。その理由は何ですか
9. 再挑戦した場合、最初のプレイから何を変えましたか
10. ほかに気になったことはありますか

聞き取りでは正解を教えたり、回答を特定の法則へ誘導したりしない。プレイヤー自身の言葉を要約しすぎず記録する。

---

## 6. 記録の分離と回収

| 記録 | 内容 | 管理方法 |
|---|---|---|
| プレイJSON | 時刻、選択履歴、手掛かり、回数、結果 | ゲームから保存したファイルを編集せず保管する |
| 記録票 | 観察、法則の仮説、理不尽さ、怖さ、ためらい、再挑戦意欲 | JSONとは別ファイルで匿名IDにより対応付ける |

- JSONのファイル名は`black-text-playtest-{session_id}.json`のまま保管する
- 記録票には対応するJSONファイル名だけを記載し、JSON全文を貼り付けない
- JSONと記録票をGitへ追加しない
- 回収には、実施前に合意した非公開の受け渡し方法を使用する
- 不要になった対応表や連絡情報は、テスト記録とは分けて削除できる状態にする

---

## 7. 問題の分類

発見した問題は、次のいずれかへ分類する。

| 分類 | 例 |
|---|---|
| ゲームルール | 手掛かりと判定が一致しない、原因を振り返れない |
| 固定文章 | 必要な事実が読み取れない、描写が矛盾する |
| UI | 選択肢を押せない、本文更新に気づけない、横スクロールが出る |
| 計測 | JSONを保存できない、選択履歴または時間が欠ける |
| 環境 | 特定ブラウザだけで表示または保存に失敗する |

分類後、`docs/07_プレイテスト記録票.md`の基準を使い、段階Aで修正する問題、段階Bへ持ち越す問題、仕様どおりで変更しない項目を区別する。

## 8. 公開環境チェックリスト

Vercelへの接続と外部公開は、ユーザーの明示的な許可と必要なアカウント権限を得てから行う。

公開時は次を確認する。

- [ ] 公開対象のコミットIDを記録した
- [ ] Lint、型チェック、単体・UIテスト、Chromium E2E、本番ビルドが成功した
- [ ] Firefox・WebKitの公開前スモークテストが成功した
- [ ] 公開範囲とURLの共有先を決めた
- [ ] 公開URLでクリア経路とゲームオーバー経路を確認した
- [ ] JSON保存と再挑戦を公開URLで確認した
- [ ] ブラウザコンソールに未処理エラーがないことを確認した
- [ ] JSONや聞き取り記録をデプロイ対象へ含めていない
