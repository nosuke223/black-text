# 04_MVP仕様

## 1. 文書の目的

本ドキュメントは、BLACK TEXTの最初のプレイ可能なMVPの範囲、進行、内部状態、判定、AIとゲームシステムの役割分担、プレイテスト項目を定義する。

本MVPは、試作シナリオ`SCENARIO-001 三度目の呼び声`を使い、次のゲームループが成立するかを確認するための縦切り試作である。

```text
文章を読む
    ↓
法則を推測する
    ↓
選択をためらう
    ↓
結果を受け取る
    ↓
失敗後に記録を振り返る
    ↓
仮説を持って再挑戦する
```

本ドキュメントは正式な汎用シナリオ生成スキーマではない。MVPで必要な最小限の契約と実装境界のみを定義する。

本ドキュメントでは、項目の状態を次のように扱う。

| 表記 | 意味 |

| 確定 | 上位資料または既存シナリオで定義済み、または本ドキュメントでレビュー・合意を経て決定したもの。変更には再レビューを要する |
| 暫定 | MVP初稿で採用する案。プレイテストまたは実装前レビューで変更可能 |

本ドキュメントは初稿であり、「暫定」の項目は実装着手前に再確認する。

## 4. MVPの目的

### 4.1 検証すること

- 文章中の手掛かりから、プレイヤーが主法則を推測できるか
- 選択肢を押す前に、迷いやためらいが生まれるか
- 一度の小さな失敗だけでクリア不能にならず、危険の蓄積が機能するか
- ゲームオーバー後の記録から、プレイヤーが原因の仮説を持てるか
- 再挑戦によって、別の選択を試したいと感じるか
- ゲームシステムが判定を固定したまま、AIが文章表現を変化させられるか

### 4.2 本MVPだけでは検証しないこと

- 複数の怪異・法則の再利用性
- 最初の3つの物の全分岐
- シナリオの自動構築
- 長期運用時のAI生成コスト
- 販売形態、課金、配信プラットフォーム
- 周回をまたぐプレイヤー記憶

## 5. MVPの基本仕様

| 項目 | 内容 | 状態 |
|---|---|---|
| 対象シナリオ | `SCENARIO-001 三度目の呼び声` | 確定 |
| 舞台 | 営業を終えた古い宿 | 確定 |
| 言語 | 日本語のみ | 確定 |
| 実行環境 | PCのブラウザを優先するWebアプリ | 確定 |
| 想定プレイ時間 | 15〜20分 | 確定 |
| 選択回数 | MVPのプレイ中は10〜11回。シナリオ全体では導入の物選択を含め11〜12回 | 確定 |
| 最初の物 | 錆びた鈴を選択済みの状態から開始 | 確定 |
| エンディング | 基本クリア1種、ゲームオーバー | 暫定 |
| プレイ履歴 | 現在の1プレイ分のみ保持 | 暫定 |
| 再挑戦 | 結果画面から初期状態で再開 | 暫定 |
| ゲームの判定 | すべてゲームシステムが実行 | 確定 |
| AIの役割 | 確定済みの出来事、手掛かり、選択肢の文章表現 | 確定 |

### 5.1 実装段階

MVPは次の2段階で実装する。

#### 段階A：固定文章による動作試作

- シーン文章と選択肢は固定文を使用する
- シーン遷移、内部状態、クリア・ゲームオーバー判定を先に検証する
- AIの出力差とゲームルールの問題を切り分けられる状態にする

#### 段階B：AIによる表現変化

- 段階Aと同じシーンID、選択肢ID、状態更新、判定を使用する
- AIは文章と選択肢の言い回しだけを変化させる
- AI出力が不正、タイムアウト、または利用不可能の場合は固定文章へ切り替える

暫定で、段階AはMVPの必須の前半とする。段階Bまで完了した状態を、BLACK TEXTの生成AIを含むMVPの完了とする。

### 5.2 実行環境

- MVPはブラウザ向けWebアプリとして実装する
- PCの主要なモダンブラウザを優先し、ChromeとEdgeを主対象とする
- FirefoxとSafari相当の動作は、公開プレイテスト前に確認する
- スマートフォン向けのレイアウト最適化、タッチ専用演出、ネイティブアプリ化はMVP対象外とする
- ローカル開発環境と、第三者がアクセスできるプレビュー環境の両方で実行可能にする

この決定はMVPの実行環境を定めるものであり、完成版の配信形態をWebだけに限定するものではない。

### 5.3 技術スタック

選定日は2026年7月16日とする。依存パッケージの正確なバージョンは初期構築時の`package-lock.json`で固定し、メジャーバージョンの更新はテスト通過を確認してから行う。

| 項目 | 採用技術 | 用途 | 状態 |
|---|---|---|---|
| 開発ランタイム | Node.js 24 LTS | 開発、ビルド、テスト、サーバー処理 | 確定 |
| パッケージ管理 | npm、`package-lock.json` | 依存関係の導入と固定 | 確定 |
| 言語 | TypeScript、strictモード | ゲーム状態、選択肢ID、判定、UI、API | 確定 |
| Webフレームワーク | Next.js 16系の安定版、App Router | ブラウザUIと段階BのサーバーAPI | 確定 |
| UI | React | 画面表示とプレイヤー操作 | 確定 |
| スタイル | Global CSS、CSS Modules | 黒背景、白文字、選択肢、最小限の演出 | 確定 |
| クライアント状態 | Reactの`useReducer`と純粋なTypeScript関数 | 現在の1プレイの状態管理 | 確定 |
| サーバーAPI | Next.js Route Handlers | 段階BのAI呼び出しとフォールバック制御 | 確定 |
| AI出力の構造検証 | Zod | JSONの型、必須項目、選択肢IDの形式検証 | 確定 |
| 単体テスト | Vitest | 状態更新、シーン遷移、クリア・ゲームオーバー判定 | 確定 |
| UIテスト | React Testing Library | 本文、選択肢、結果表示の確認 | 確定 |
| E2Eテスト | Playwright | ブラウザ上の主要経路と再挑戦の確認 | 確定 |
| Lint | ESLint | TypeScriptとReactの静的検査 | 確定 |
| MVP公開先 | Vercel | プレビュー環境と公開プレイテスト | 暫定 |
| データベース | 使用しない | プレイ履歴を永続保存しないため | 確定 |

Zodで検証するのはAI出力の構造である。選択肢の意味、固定事実、手掛かりの意味が保たれているかを自動検証する方法は、引き続き段階B着手前の未決事項とする。

### 5.4 構成方針

- ゲームルール、状態更新、シーン遷移、結果判定を、ReactやAI APIへ依存しない純粋なTypeScriptとして分離する
- 段階Aでは、固定シナリオデータとブラウザ内の状態だけで最後までプレイ可能にする
- 段階Bでは、同じゲームルールを保ったまま、`/api/generate-scene`相当のRoute Handlerを追加する
- AIのAPIキーはサーバーだけが参照し、`NEXT_PUBLIC_`が付く環境変数やクライアントコードへ含めない
- AI生成が失敗した場合は、ゲームルールを変更せず固定文章へ切り替える
- 現在の1プレイの状態だけをブラウザメモリに保持し、ページ再読み込み後の復元はMVP対象外とする
- Tailwind CSS、外部状態管理ライブラリ、データベース、認証機能は追加しない

### 5.5 候補比較

| 候補 | 評価 | 判断 |
|---|---|---|
| Next.js、React、TypeScript | 段階AのブラウザUIと段階BのサーバーAPIを単一プロジェクトで扱える | 採用 |
| React、Vite、TypeScript | 段階Aは軽量だが、段階Bで別途サーバー構成を決める必要がある | 不採用 |
| SvelteKit、TypeScript | フルスタック構成は可能だが、本MVPではNext.jsより優先する要件上の利点がない | 不採用 |

## 7. プレイ進行

```text
タイトル
    ↓
ゲーム開始
    ↓
錆びた鈴を持っていることを示す導入
    ↓
違和感
    ↓
法則の推測
    ↓
圧迫
    ↓
最終局面
    ↓
クリア / ゲームオーバー
    ↓
断片的な記録
    ↓
最初から再挑戦 / 終了
```

### 7.1 最初の3つの物の扱い

MVPは「錆びた鈴を選んだ直後」から開始する。

理由は、未実装の2つの物を選択可能に見せ、プレイ中に開発用メッセージへ戻すと、恐怖体験とプレイテストを中断するためである。

このため、「3つの物から1つを選ぶ体験」は本MVPの検証対象外とする。ただし、鈴が中盤と最終局面につながることは検証する。

## 8. UI仕様

### 8.1 共通方針

- 画面は黒を基調とする
- 文字は白を基調とする
- 常時表示する要素は、本文、選択肢、必要最小限の操作に限る
- 怪異圧力、法則違反などの内部状態は表示しない
- 内部状態の変化は、文章、表示タイミング、選択肢数などで示唆する
- 選択肢を1つ選んだ後は、次のシーンが確定するまで再選択できない
- 進行先がある選択肢0件のシーンでは「次へ」を表示する。「次へ」は`ChoiceId`を持たず、その操作自体は選択回数、`choice_history`、シナリオ固有状態へ寄与しない。ただし、遷移先シーンに定義された入場時更新は通常どおり適用する
- 選択回数を増やすためだけの無意味な操作は追加しない

### 8.2 画面

| 画面 | 必要な要素 |
|---|---|
| タイトル | プロジェクト名、開始操作 |
| プレイ | シーン本文、0〜4個の選択肢、0件の場合の「次へ」操作 |
| 結果 | クリアまたはゲームオーバーの短い文章 |
| 記録 | 選択した物、確認した手掛かり、特徴的な回数、最後の選択、プレイテスト記録の保存操作 |
| 再挑戦 | 最初からやり直す操作、終了操作 |

タイプライター表示、音、ノイズ、点滅などの具体的な演出は未決とする。まずは演出がなくても全文を読み、選択できる状態を成立させる。

## 9. ゲーム状態

ゲーム状態は、全シナリオで必要な共通状態と、`SCENARIO-001`のシナリオ固有状態に分ける。

`docs/02_ゲームデザイン.md`では危険度と怪異圧力を別の候補としているが、本MVPでは独立した危険度を持たない。返事や鈴の使用による危険の蓄積は、`entity_pressure`へ統合して表現する。

### 9.1 共通状態

| 状態 | 初期値 | 用途 |
|---|---|---|
| `scene_id` | 導入シーン | 現在のシーン |
| `selected_item` | `rusty_bell` | 最初に選んだ物。MVPでは固定 |
| `choice_history` | 空 | シーンID、選択肢ID、表示文、選択順序を保持 |
| `outcome` | `playing` | `playing` / `clear` / `game_over` |

### 9.2 シナリオ固有状態

| 状態 | 初期値 | 更新条件 | 用途 |
|---|---:|---|---|
| `call_count` | 0 | 直接的な呼びかけが発生するたびに+1 | 三度目の判定 |
| `response_count` | 0 | 声へ返事・問いかけをするたびに+1 | 模倣精度と振り返り |
| `bell_uses` | 0 | 鈴を鳴らすたびに+1 | 鈴の効果と副法則の判定 |
| `entity_pressure` | 0 | 返事、鈴、危険行動で上昇 | 文章演出とゲームオーバー補助 |
| `clues` | 空 | 手掛かりを確認すると追加 | 最終判定と振り返り |
| `fatal_violation` | `false` | 三度目の呼びかけに返事すると`true` | 即時ゲームオーバー |

`entity_pressure`は0〜5の整数とし、5を超える更新は5に丸める。

```text
0：未認識
1：気配
2：接近
3：干渉
4：侵入直前
5：侵入
```

現行のシーン構成を初期状態から進行した場合、致命的な主法則違反を除く通常の選択による`entity_pressure`の到達最大値は4となる。これは更新上限を4へ制限する規則ではない。通常の危険蓄積は、最終局面で`entity_pressure >= 3`になったときの直接脱出失敗として判定する。

`entity_pressure >= 5`のゲームオーバー判定は、致命的な主法則違反と、将来の状態更新で侵入段階へ達した場合を取りこぼさないための安全網として残す。

これらの状態はゲームシステムだけが更新する。AIは状態値を生成、修正、推測しない。

## 10. MVP用シナリオ契約

### 10.1 プレイ開始前に固定する事実

- プレイヤーは二階の客室付近から開始する
- 出口は一階の玄関にある
- 怪異は声を利用して返事を引き出す
- 返事や音による反応で怪異圧力が上がる
- 三度目の直接的な呼びかけへの返事は致命的な法則違反である
- 錆びた鈴は怪異がいる方向を示す
- 鈴を三回以上鳴らすと、怪異にプレイヤーの位置を特定される
- 最終局面では玄関の向こうから安全を装う声が聞こえる
- 実際の怪異はプレイヤーの背後にいる
- シナリオの途中で上記の事実を変更しない

### 10.2 手掛かり

| ID | 強さ | 正準的な意味 |
|---|---|---|
| `CLUE-A` | 弱 | 呼びかけの回数に意味がある |
| `CLUE-B` | 中 | 三度目の声は人ではない |
| `CLUE-C` | 強 | 三度目には返事をしない |
| `CLUE-D` | 弱 | 鈴の使用には代償がある |
| `CLUE-E` | 弱 | 怪異はプレイヤーの反応を材料に模倣する |

AIは手掛かりの文体を変えてよいが、正準的な意味を変更しない。

手掛かりは、正準的な意味を含む本文の表示が保証された選択結果と一体で`clues`へ追加し、本文を見せずにフラグだけを先行させない。`S06`の選択前には`CLUE-B`の正確な文を見せず、宿帳を詳しく調べた場合だけ`CLUE-B`と`CLUE-D`を表示・追加する。

## 11. シーン構成

現時点の既存シナリオに定義されているシーンを、MVPの実装単位として整理する。

| シーンID | 役割 | 呼びかけ | 選択肢数 | 主な更新 |
|---|---|---:|---:|---|
| `P00` | 錆びた鈴を持った状態を示す導入 | - | 0 | `selected_item = rusty_bell` |
| `S01` | 最初の声 | 1回目 | 2 | `call_count`、返事時の圧力 |
| `S02` | 廊下の見方 | - | 2 | 行動履歴 |
| `S03` | 破れた張り紙 | - | 2 | `CLUE-A` |
| `S04` | 二度目の声 | 2回目 | 3 | 返事または鈴の使用 |
| `S05` | 二度目の声の記録 | - | 2 | `CLUE-E`、行動履歴 |
| `S06` | 宿帳 | - | 2 | 詳細確認時に`CLUE-B`、`CLUE-D` |
| `S07` | 玄関までの道 | - | 2 | 行動履歴 |
| `S08` | 三度目の声 | 3回目 | 3 | 致命的違反、鈴の使用 |
| `S09` | 最後の日記 | - | 0 | `CLUE-C` |
| `S10` | 最後の確認 | - | 2 | 主法則または鈴の使用履歴の振り返り |
| `S11` | 玄関 | - | 3 | 直接脱出、鈴、振り返り |
| `S12` | 鈴で背後を確認後 | - | 2 | 脱出または振り返り |
| `R_CLEAR` | クリア結果 | - | 0 | `outcome = clear` |
| `R_GAME_OVER` | ゲームオーバー結果 | - | 0 | `outcome = game_over` |

### 11.1 選択回数と小さな選択

`SCENARIO-001`は、導入の物選択を含め11〜12回の選択で構成する。MVPでは錆びた鈴を選択済みの状態から開始するため、プレイ中の選択は10〜11回となる。

追加した4回の小さな選択は、次の役割を持つ。

- `S02`：探索と後続描写に使う行動履歴
- `S05`：怪異の模倣に関する弱い手掛かりと観察履歴
- `S07`：三度目の呼びかけ前のリズムと経路履歴
- `S10`：最終局面前の主法則または鈴の使用履歴の確認

どの小さな選択も、回数を増やすためだけの操作にしない。

## 12. 選択肢IDと内部分類

AIに表示文を変化させる場合も、選択肢IDと意味は変更しない。

| シーン | 選択肢ID | 内部分類 | 意味 |
|---|---|---|---|

## 13. 状態更新と判定順序

選択後は、必ず次の順序で処理する。

1. 選択した選択肢IDを`choice_history`へ追加する
2. 選択肢IDに対応する内部状態を更新する
3. 即時ゲームオーバー条件を判定する
4. クリア条件を判定する
5. 結果が確定していない場合のみ次のシーンへ進む
6. 確定済みの結果と状態をもとに表示文を作る

AIの文章生成を状態更新や結果判定より先に実行しない。

### 13.1 呼びかけと返事

- `S01`、`S04`、`S08`の表示前に`call_count`を1増やす
- `RESPONSE`を選ぶと`response_count`と`entity_pressure`を1増やす
- `S08_REPLY`を選ぶと`fatal_violation = true`、`entity_pressure = 5`とし、ゲームオーバーにする

### 13.2 鈴の使用

- `BELL_USE`を選ぶと、先に`bell_uses`と`entity_pressure`を1増やす
- 更新後の`bell_uses >= 3`で、怪異に位置を特定されたものとし、MVPではゲームオーバーにする
- 更新後の`bell_uses <= 2`であれば、シーンごとに固定された反響方向を提示する
- `S04_RING_BELL`の反響方向は、階段とは反対方向に固定する
- `S08_RING_BELL`の反響方向は、`S07`で選んだ経路の背後、今しがた通ってきた方向に固定する
- `S11_RING_BELL`の反響方向は、プレイヤーの背後に固定する
- `S11_RING_BELL`を安全に使用できた場合は`S12`へ進む

### 13.3 怪異圧力

- 状態更新後に`entity_pressure >= 5`であれば、ほかの判定より先にゲームオーバーにする
- `entity_pressure`の段階によって文章の乱れ、声の模倣精度、選択肢の圧迫感を変化させる
- 圧力の段階が同じなら、同じ選択肢IDに対する判定を変えない

### 13.4 手掛かりの取得

- 対応する本文の表示と手掛かりフラグの追加は、一つの選択結果として不可分に扱う
- AIによる本文が検証に失敗した場合は固定文を表示し、その表示を確定してからフラグを追加する
- `S06`の選択前は、`CLUE-B`の末尾を伏せた文章だけを表示する
- `S06_INSPECT_BOOK`では、`CLUE-B`と`CLUE-D`の本文を表示してから両方を`clues`へ追加する
- `S06_CLOSE_BOOK`では、`CLUE-B`と`CLUE-D`の本文を表示せず、どちらも追加しない

## 14. クリア条件

### 14.1 鈴を使って方向を確認する場合

次の条件をすべて満たし、`S12_OPEN_DOOR`を選ぶとクリアにする。

- `fatal_violation == false`
- `bell_uses <= 2`
- `entity_pressure < 5`
- `S12_LOOK_BACK`を選んでいない

### 14.2 鈴を使わずに玄関を開ける場合

既存シナリオに合わせ、次の条件をすべて満たし、`S11_OPEN_DOOR`を選ぶとクリアにする。

- `fatal_violation == false`
- `entity_pressure <= 2`
- `CLUE-B`を発見済み

`S11_OPEN_DOOR`を選んだ時点で上記のクリア条件を満たさない場合は、追加の危険ルートを増やさずゲームオーバーにする。

## 15. ゲームオーバー条件

次のいずれかを満たした場合、ゲームオーバーにする。

| 条件 | 判定 |
|---|---|
| 三度目の直接的な呼びかけに返事する | `fatal_violation == true` |
| 鈴を三回以上使用する | 更新後の`bell_uses >= 3` |
| 怪異圧力が侵入段階に達する | `entity_pressure >= 5` |
| 最終局面で背後を振り返る | `S11_LOOK_BACK`または`S12_LOOK_BACK` |
| 玄関を直接開けたが、クリア条件を満たしていない | `S11_OPEN_DOOR`選択時の条件不成立 |

同じ状態と同じ選択肢IDの組み合わせであれば、常に同じ判定を返す。

### 15.1 最終局面で安全策を使い切った状態

`bell_uses == 2`であり、かつ（`CLUE-B`を発見していない、または`entity_pressure >= 3`である）状態で`S11`へ到達すると、3つの選択肢はすべてゲームオーバーになる。

- `S11_OPEN_DOOR`：直接脱出の条件を満たさない
- `S11_RING_BELL`：三回目の鈴使用になる
- `S11_LOOK_BACK`：最終局面で背後を振り返る

この状態は進行不能ではなく、それ以前の手掛かり確認、返事、鈴の使用が積み重なった終局として意図的に採用する。`S11`へ到達するまでに怪異圧力の高まりを文章で示唆し、結果後の記録から原因の仮説を持てることをプレイテストで確認する。

## 16. 結果後の記録

ゲームオーバー後は正解や死因を完全に説明せず、次の情報を断片的に表示する。

- 選択した物：錆びた鈴
- 確認した手掛かりの文章
- 声に返事した回数
- 鈴を鳴らした回数
- 最後に選んだ選択肢の表示文

次の内部情報はそのまま表示しない。

- `fatal_violation`
- `entity_pressure`の数値
- ゲームオーバー条件の正式名
- クリアに必要な正確な条件式

クリア後も、ゲームオーバー時と同じ項目を記録として表示してよい。

## 17. AIとゲームシステムの役割分担

### 17.1 ゲームシステムが管理するもの

- シーンID
- 選択肢IDと内部分類
- 選択肢数
- 内部状態の更新
- 手掛かりの正準的な意味
- シーン遷移
- クリア判定
- ゲームオーバー判定
- AIに渡す事実と禁止事項
- AI出力の検証とフォールバック

### 17.2 AIに任せるもの

- 古い宿の具体的な描写
- 声の主を誰に似せるか
- 声の模倣精度が上がる表現
- 固定済みの手掛かりの自然な文体
- 選択肢の自然な言い回し
- 過去の選択を利用した心理的な揺さぶり
- 怪異圧力に応じた文章の乱れ
- 判定済みのゲームオーバーに対する短い恐怖文

### 17.3 AIに任せないもの

- 主法則の生成・変更
- 呼びかけ回数の判定
- 選択肢IDの追加・削除・変更
- 内部状態の更新
- 手掛かりの意味の変更
- 手掛かりの表示可否と取得条件の変更
- クリア・ゲームオーバー判定
- 怪異が最終局面で背後にいるという事実の変更
- 過去の選択履歴の改変

## 18. MVP用のAI入出力契約

この契約は`SCENARIO-001`のMVPのための最小形であり、正式な汎用JSONスキーマではない。

### 18.1 AIへ渡す情報

- `scene_id`
- シーンで必ず起きる出来事
- 文中に含める必須事実
- 表示する選択肢IDと各選択肢の意味
- 表示可能な手掛かりとその正準的な意味
- 怪異圧力の段階
- 文章表現に必要な過去の選択だけを抽出した履歴
- 変更してはいけない事実
- 後続シーンの結論を先に漏らさないための禁止事項

### 18.2 AIから受け取る情報

```json
{
  "scene_id": "S01",
  "scene_text": "シーン本文",
  "options": [
    {
      "id": "S01_REPLY",
      "text": "声に返事をする"
    },
    {
      "id": "S01_SILENCE",
      "text": "黙って廊下へ出る"
    }
  ]
}
```

### 18.3 出力検証

ゲームシステムは、AI出力に対して最低限次を確認する。

- `scene_id`が指定したシーンと一致する
- `scene_text`が空でない
- 指定した選択肢IDがすべて含まれる
- 未指定の選択肢IDが追加されていない
- 選択肢IDが重複していない
- 選択肢数がゲームシステムの指定と一致する
- 各選択肢の表示文が、指定した内部分類の意味を反転または変更していない
- シーンで必要な固定事実が欠落していない
- 現在は表示できない手掛かりの正準的な意味が含まれていない

選択肢の意味と固定事実の自動検証方法は、段階Bの着手前に決定する。

検証失敗時は再判定をAIに任せず、そのシーンの固定文章と固定選択肢を表示する。

## 19. プレイテスト

### 19.1 確認項目

- プレイヤーは呼びかけの回数を数えられるか
- `CLUE-A`だけでも回数を意識できるか
- `CLUE-B`または`CLUE-C`から主法則を推測できるか
- 鈴の利点と代償が理解できるか
- 返事を1回しただけでクリア不能になっていないか
- 鈴を二回使って最終局面の安全策を失った場合、事前の兆候と結果後の記録から蓄積した原因を推測できるか
- 最終局面の判断が、過去の手掛かりと行動につながっているか
- 選択肢の前で、読み返しやためらいが発生するか
- ゲームオーバー後の記録から、原因の仮説を持てるか
- ゲームオーバー後に再挑戦したいと感じるか
- 15〜20分で終了するか
- AIが文章を変えても、手掛かりの意味と判定が変わらないか

### 19.2 記録する情報

- プレイ時間
- 各選択肢の表示から選択までの時間
- 選択した選択肢ID
- 発見した手掛かり
- クリアまたはゲームオーバーの結果
- プレイ後にプレイヤーが説明した法則の仮説
- 理不尽に感じた箇所
- 怖さ、ためらい、再挑戦意欲に関する感想

初回プレイテスト前に、合格率などの数値目標を確定しない。まずは少数のプレイから問題の種類を抽出し、その後に必要な定量指標を決める。

### 19.3 段階Aでの記録方法

自動計測できる情報は、現在の1プレイ分だけをブラウザメモリに保持し、結果画面からJSONファイルとして保存できるようにする。

JSONには、最低限次の情報を含める。

- 記録形式のバージョン
- ゲーム仕様と判定ルールを識別する`game_version`
- プレイを識別するランダムなセッションID
- プレイ開始・終了時刻と総プレイ時間
- 選択順、シーンID、選択肢ID、表示した選択肢文、選択までの時間
- 発見した手掛かり
- 返事をした回数
- 鈴を使用した回数
- クリアまたはゲームオーバーの結果
- 最後に選んだ選択肢ID

段階Aの初期実装では、記録形式を`schema_version = 1.0.0`、判定ルールを`game_version = scenario-001-rules-v1`として識別する。記録形式または判定結果の再現性へ影響する変更を行った場合は、対応するバージョンを更新する。

「16. 結果後の記録」でプレイヤーへ直接表示しないとしている`entity_pressure`、`fatal_violation`、正式なゲームオーバー条件名はJSONにも含めない。氏名、メールアドレス、IPアドレスなどの個人情報も収集しない。

プレイテスト分析で内部状態の推移が必要な場合は、JSONに記録した`game_version`と一致するゲームエンジンへ選択履歴を再入力して再計算する。再計算した内部状態は、プレイヤー向けの記録画面やJSONには追加しない。

プレイヤーが説明した法則の仮説、理不尽に感じた箇所、怖さ、ためらい、再挑戦意欲は自動計測せず、プレイ後の聞き取りまたは別紙で記録する。

データベースへの保存、自動アップロード、外部分析サービスへの送信は段階Aでは行わない。

- すべての選択肢IDに後続シーンまたは結果がある

- クリア不能になる選択を行った場合、それ以前に警告または兆候がある
- 固定文章版とAI表現版で、同じ選択肢IDの履歴から同じ結果に到達する
