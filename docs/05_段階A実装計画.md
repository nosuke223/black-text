# 05_段階A実装計画

## 1. 目的

本ドキュメントは、`docs/04_MVP仕様.md`で定義した段階A「固定文章による動作試作」を、実装可能な作業単位へ分解する。

段階Aでは、`SCENARIO-001 三度目の呼び声`を固定文章で最後までプレイできる状態を作り、次を検証する。

- シーンIDと選択肢IDに基づいて進行できる
- 内部状態、手掛かり、履歴を一貫して更新できる
- クリア・ゲームオーバーをゲームシステムだけで判定できる
- 結果後の記録と再挑戦が機能する
- プレイテストに必要な操作データを回収できる
- AIを追加する前に、ゲームルールとUIの問題を切り分けられる

本ドキュメントは実装順と受け入れ条件を定める。ゲーム内容または判定が`docs/04_MVP仕様.md`と衝突する場合は、`docs/04_MVP仕様.md`を正とする。

## 4. 実装境界

### 4.1 ゲーム領域

ゲーム領域はReact、Next.js、ブラウザAPI、AI APIへ依存させない。

ゲーム領域が管理するもの：

- シーンIDと選択肢ID
- ゲーム状態
- 選択履歴
- 状態更新
- シーン遷移
- 手掛かりの取得
- クリア・ゲームオーバー判定
- 結果後に表示する記録

同じ初期状態と同じ選択肢ID列を与えた場合は、常に同じ結果と状態を返す。

### 4.2 UI

UIはゲーム領域から受け取った本文と選択肢を表示し、選択肢IDをゲーム領域へ渡す。

UIは内部状態を直接書き換えず、独自のクリア・ゲームオーバー判定を持たない。

`P00`と`S09`のように選択肢がないシーンでは、UI専用の「次へ」操作を表示する。「次へ」は`ChoiceId`を持たず、その操作自体は選択回数、`choice_history`、シナリオ固有状態へ寄与しない。ただし、遷移先シーンに定義された入場時更新は通常どおり適用する。

### 4.3 プレイテスト記録

計測処理はゲーム結果へ影響させない。計測に失敗しても、プレイを継続できるようにする。

記録は現在の1プレイ分だけをブラウザメモリに保持し、外部へ自動送信しない。

## 5. 想定フォルダ構成

初期構築時は、次の責務分割を基本とする。実装上の必要が確認できるまでは、階層を増やさない。

```text
src/
├─ app/
│  ├─ layout.tsx
│  ├─ page.tsx
│  └─ globals.css
├─ components/
│  └─ game/
├─ game/
│  ├─ domain/
│  ├─ scenarios/
│  │  └─ scenario-001/
│  └─ playtest/
tests/
└─ e2e/
```

Next.jsアプリのルートはリポジトリ直下とし、`package.json`、`src/`、`tests/`を`docs/`と同じ階層へ置く。`web/`などの入れ子のアプリディレクトリは作らない。

単体テストとUIテストは、対象ファイルの近くに`*.test.ts`または`*.test.tsx`として配置する。E2Eテストは`tests/e2e/`へ配置する。

### A1. ゲーム領域の型と初期状態

#### 作業

- `SceneId`、`ChoiceId`、`ClueId`、`Outcome`を文字列リテラル型として定義する
- `GameState`へ`scene_id`、`selected_item`、`choice_history`、`outcome`とシナリオ固有状態を定義する
- `call_count`、`response_count`、`bell_uses`、`entity_pressure`、`clues`、`fatal_violation`を必須項目にする
- `S07`で選んだ経路を後続描写へ渡すための履歴を定義する
- `ChoiceId`と実際に表示した選択肢文を持つ`ChoiceSelection`を定義する
- 選択と「次へ」を区別する`GameAction`を定義する
- 選択順、シーンID、選択肢ID、表示文を保持する`ChoiceHistoryEntry`を定義する
- 初期状態を新しく生成する`createInitialGameState`を用意する
- 再挑戦時にすべての状態と配列を新しい値へ戻せるようにする

#### 受け入れ条件

- 型定義から`SCENARIO-001`で使用しない汎用状態を増やしていない
- 初期状態が`docs/04_MVP仕様.md`の`9.1`、`9.2`と一致する
- 状態型と初期状態がReactやブラウザAPIをimportしていない
- 初期化と再初期化の単体テストが成功する

### A2. 固定シナリオデータ

#### 作業

- `P00`、`S01`〜`S12`、`R_CLEAR`、`R_GAME_OVER`を型付きデータとして定義する
- `docs/03_恐怖設計/試作シナリオ/001_三度目の呼び声.md`の固定文章を実装用データへ移す
- `docs/04_MVP仕様.md`の選択肢ID、内部分類、意味を定義する
- 各シーンの選択肢数を仕様と一致させる
- `S06`の選択前には`CLUE-B`の正確な文を表示しない
- `S06_INSPECT_BOOK`の結果でのみ`CLUE-B`と`CLUE-D`の本文を表示可能にする
- `S08_RING_BELL`の反響方向を、`S07`で選んだ経路の背後に固定する
- 結果画面と記録画面に使う固定文章を定義する

#### 受け入れ条件

- すべてのシーンIDが一意である
- すべての選択肢IDが一意である
- 各シーンの選択肢数と定義件数が一致する
- すべての選択肢IDに内部分類と表示文がある
- 未取得の手掛かりを先に表示するデータがない
- シナリオデータがAI、乱数、時刻に依存していない

### A3. 状態更新・遷移・結果判定

#### 作業

- `GameState`と`GameAction`から次の状態を返す純粋関数を実装する
- 選択の判定には`ChoiceSelection`の`ChoiceId`だけを使用し、表示文は履歴記録だけに使用する
- `S01`、`S04`、`S08`へ入る前に`call_count`を増やす
- 返事、鈴、手掛かり、経路、履歴の更新を選択肢IDへ対応付ける
- 鈴使用時は`bell_uses`と`entity_pressure`を更新してから判定する
- `entity_pressure`を0〜5へ制限する
- 即時ゲームオーバー、クリア、次シーンの順序を仕様どおりに判定する
- `S06`の本文表示と手掛かり追加を一つの選択結果として扱う
- `S09`の本文表示が確定した時点で`CLUE-C`を追加し、「次へ」で二重に追加しない
- `S11`と`S12`の直接脱出、鈴使用、振り返りを固定判定する
- `bell_uses == 2`かつ安全な直接脱出条件がない状態では、`S11`の全選択肢がゲームオーバーになることを固定する
- `P00`と`S09`のUI専用「次へ」自体は選択回数、選択履歴、シナリオ固有状態へ含めず、遷移先シーンの入場時更新は通常どおり適用する
- 不正なシーンIDまたは選択肢IDを受け取った場合は、状態を壊さず明示的なエラーにする

#### 受け入れ条件

- 同じ初期状態と選択肢ID列から常に同じ最終状態になる
- React、Next.js、DOM、ネットワーク、乱数、現在時刻へ依存していない
- `docs/04_MVP仕様.md`の`20.1`にある段階A対象の判定テストがすべて成功する
- すべての選択肢に次シーンまたは結果がある
- 進行不能な未定義状態がない

### A4. 固定文章版UI

#### 作業

- タイトル画面からゲームを開始できるようにする
- プレイ画面に本文と2〜4個の選択肢を表示する
- 選択時は`ChoiceId`と画面に表示した選択肢文を`ChoiceSelection`としてゲーム領域へ渡す
- `P00`と`S09`では、選択肢ではない「次へ」操作を表示する
- 選択後、次のシーンが確定するまで選択肢を再操作できないようにする
- 結果画面にクリアまたはゲームオーバーの短い文章を表示する
- `docs/04_MVP仕様.md`の`16`で非表示としている内部判定値、正式な死因、正確なクリア条件を表示しない
- 黒背景、白文字、必要最小限の余白とフォーカス表示を実装する
- ボタンをキーボードで操作できるようにする
- シーン更新後に本文へフォーカスを移すか、支援技術へ更新を通知する
- 段階Aでは本文を即時表示し、タイプライターや自動進行を追加しない

#### 受け入れ条件

- タイトルから結果まで、マウスとキーボードのどちらでも進行できる
- 選択肢IDではなく自然な日本語だけがプレイヤーへ表示される
- 怪異圧力、致命的違反フラグ、正式なゲームオーバー条件名が画面へ表示されない
- 二重選択で履歴または内部状態が二回更新されない
- PC幅で本文と選択肢が読みやすく、横スクロールが発生しない
- UIテストで各画面と主要な状態変化を確認できる

---

### A5. 記録・JSON保存・再挑戦

#### 作業

- プレイ開始時にランダムなセッションIDと開始時刻を生成する
- 各選択肢の表示時刻と選択時刻から`decision_time_ms`を記録する
- 結果画面で総プレイ時間を確定する
- `docs/04_MVP仕様.md`の`19.3`に従うJSONデータを生成する
- ゲーム仕様と判定ルールを識別する`game_version`を定義し、JSONへ含める
- `black-text-playtest-{session_id}.json`としてブラウザから保存できるようにする
- JSONへ`entity_pressure`、`fatal_violation`、正式なゲームオーバー条件名、個人情報を含めない
- 内部状態の分析が必要な場合は、記録された`game_version`と一致するゲームエンジンへ選択履歴を再入力して再計算する
- 記録画面に、確認した手掛かり、返事回数、鈴使用回数、最後の選択を表示する
- 再挑戦でゲーム状態と計測状態を初期化し、新しいセッションIDを発行する
- 計測またはJSON保存に失敗しても、結果表示と再挑戦を妨げない

セッションIDの乱数は記録の識別だけに使用し、シーン、選択肢、状態更新、結果判定へ渡さない。

#### 受け入れ条件

- JSONに`schema_version`と`game_version`がある
- 各選択の`decision_time_ms`が0以上である
- 選択順と`choice_history`が一致する
- JSONの選択履歴を同じ`game_version`のゲームエンジンへ再入力すると、プレイ終了時の内部状態を再現できる
- JSON保存時に外部ネットワーク通信が発生しない
- 再挑戦後に前回の状態と記録が残らない
- JSON生成と再挑戦の単体テストまたはUIテストが成功する

### A6. 結合確認・E2E・品質確認

#### 必須E2E経路

- `CLUE-B`を取得し、低い怪異圧力で直接脱出してクリアする
- 鈴を二回以内で使い、背後を見ずに脱出してクリアする
- 三度目の呼びかけに返事をしてゲームオーバーになる
- 鈴を三回使ってゲームオーバーになる
- 最終局面で振り返ってゲームオーバーになる
- `CLUE-B`未取得または怪異圧力3以上で直接脱出に失敗する
- 最終局面で安全策を使い切り、どの選択でもゲームオーバーになる
- 結果画面からJSONを保存する
- 再挑戦で初期状態へ戻る

#### 品質確認

- `npm run lint`
- `npm run typecheck`
- `npm run test:run`
- `npm run test:e2e`
- `npm run build`

ChromiumのE2Eを必須とする。FirefoxとWebKitは、第三者向けプレビューを開始する前に主要経路のスモークテストを行う。

#### 受け入れ条件

- 必須コマンドがすべて成功する
- 必須E2E経路がすべて成功する
- ブラウザのコンソールに未処理エラーがない
- 固定文章と判定が`docs/04_MVP仕様.md`から逸脱していない

## 9. テストの責務

| テスト層 | 主な対象 | 扱わないもの |
|---|---|---|
| 単体テスト | 状態更新、遷移、判定、JSON生成 | DOMの見た目、実ブラウザ操作 |
| UIテスト | 本文、選択肢、結果、記録、再挑戦 | 複数画面を通した全経路 |
| E2Eテスト | タイトルから結果までの主要経路、JSON保存 | 内部関数の細かな分岐 |
| 手動確認 | 読みやすさ、ためらい、怖さ、ブラウザ差 | 判定の網羅性 |

単体テストで判定を網羅し、E2Eテストで同じ分岐を過剰に重複させない。E2Eはプレイヤー体験として重要な経路へ絞る。
