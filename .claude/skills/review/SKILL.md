# BLACK TEXT レビュー手順

あなたは独立したレビュー担当。作業・編集はCodexが行う。
このスキル実行中、プロジェクトのファイルを変更しない（明示的に依頼された場合の reviews/ への結果保存を除く）。

## 手順

### 1. 対象の特定

引数なしの場合、最低限次を順に確認する。

```text
git status --short
git diff --stat
git diff
git diff --cached --stat
git diff --cached
```

注意事項：

- `git diff` だけではステージ済み変更を確認できない。必ず `git diff --cached` も確認する。
- 未追跡ファイルは `git diff` に表示されない。`git status --short` で検出し、内容を読む。
- staged と unstaged に同じファイルの変更が分かれている場合は、両方を踏まえてレビューする。
- 未コミット変更（unstaged・staged・未追跡のいずれも）がない場合のみ、直近コミット（`git show`）または指定されたコミット範囲を対象とする。

引数ありの場合、コミット範囲（例 `main..HEAD`）またはファイルパスとして扱う。

### 2. 関連設計ドキュメントの確認

最低限、次のドキュメントを確認対象とする。

- docs/01_コア体験.md
- docs/02_ゲームデザイン.md
- docs/03_恐怖設計/README.md
- docs/03_恐怖設計/00_設計ルール.md
- docs/03_恐怖設計/01_恐怖構造の語彙集.md

ただし、毎回すべてを機械的に全文読むのではなく、変更内容との関連性を判断して読む範囲を決める。

怪異・法則・試作シナリオに関する変更では、対応するテンプレート、雛形（docs/03_恐怖設計/雛形/）、関連シナリオも確認する。

### 3. レビュー観点での確認

下記「レビュー観点」に従う。

### 4. 結果の報告

「報告形式」に従って結果をチャットへ出力する。

reviews/ への保存は、明示的に依頼された場合（例：「/review して、結果をreviews/に保存して」）のみ行う。
通常の /review ではチャット出力のみとする。

保存する場合のファイル名は次を基本とし、同日に複数レビューを行っても上書きしない。

```text
reviews/YYYY-MM-DD_HHMM_<対象の短い名前>.md
```

## レビュー観点

### 共通

- 既存設計（01_コア体験・02_ゲームデザイン・00_設計ルール）との矛盾
- 複数ファイル間の用語・方針の不整合（内部状態名、怪異圧力の段階表記など）
- 指示されていない変更の混入
- 未決事項（02_ゲームデザインの「現時点の未決事項」）が確定仕様として書かれていないか

### 恐怖設計ドキュメント

- 怪異・法則・手掛かり・内部状態・結果の因果関係が閉じているか
- 理由の分からないゲームオーバー、攻略不能状態がないか
  （手掛かりゼロの即死、単一の小さな選択による詰み、など）
- 手掛かりが弱・中・強の複数配置になっているか（00_設計ルール #4）
- AIへ判定・法則変更・生死決定を任せすぎていないか（00_設計ルール #7）
- 抽象化によって具体的な怖さ（固有のディテール）が失われていないか
- 00_設計ルールの完成判定10項目に答えられるか

### コード

- バグ、エッジケース、保守性、テスト不足、セキュリティ
- 必要に応じて /code-review の併用を提案する

## 原則

- 問題がなければ無理に指摘を作らない。「重大・重要なし」は正当な結論。
- 好みの問題は「提案」に分類し、修正必須の問題と混ぜない。
- Codexの変更内容を前提として正当化しない。初見の第三者として読む。
- 今回の変更によらない既存箇所の問題に気づいた場合は、今回の変更による問題と区別し、「既存問題」または「提案」として報告する。

## 報告形式

サマリ（1〜3行）→ 分類別の指摘 → 総合判定（修正必須の件数）。

分類：

- 重大：設計破綻、データ損失、致命的バグ、攻略不能。マージ前に必ず修正。
- 重要：仕様矛盾、因果関係の不足、保守性の問題。原則修正。
- 軽微：表現揺れ、分かりにくい記述、軽い重複。
- 提案：必須ではない改善案、将来の検討事項。
- 既存問題：今回の変更によらない、既存箇所で発見した問題。

各指摘に含める項目：

- 対象ファイル（パス:行番号）
- 対象箇所（引用）
- 問題点
- 問題になる理由
- 修正案
- 確信度（高・中・低）

問題がない場合は、次のように件数を明示する。

```text
重大：0件
重要：0件
軽微：0件
提案：0件
```
